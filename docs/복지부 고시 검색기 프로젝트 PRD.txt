복지부 고시 검색기 프로젝트 PRD

1단계. 수집범위는 [고시]만을 대상으로 한다. 
2단계. 복지부 게시판의 페이지를 1페이지부터 끝페이지까지 순회하면서 제·개정 구분, 발령번호, 제목, 등록일, 미리보기, 상세 URL을 저장한다. 수집은 Python + requests + BeautifulSoup로 시작한다. 
3단계. 목록 이외에도, 각 상세 페이지에 들어가서 제·개정일, 발령번호, 첨부파일 링크, 본문 요약 같은 추가 메타데이터를 저장해야 한다. 복지부는 또한 “행정규칙 전문은 국가법령정보센터에서 확인”하도록 연결하므로 메타데이터 저장을 잘 해야 한다. 
4단계. 의료고시의 핵심은 종종 게시판 본문보다 붙임 PDF/HWP/HWPX의 별표·별지에 있으므로, 목록/상세 수집과 첨부 수집은 분리해야 한다. 먼저 첨부파일 URL과 파일명만 저장해 두고, 2차 작업에서 PDF부터 텍스트 추출하고 HWP/HWPX는 후순위로 처리하면 된다. 초기 검색기는 첨부 원문 링크만 제공해도 충분히 쓸 수 있고, 이후 검색 품질을 높이기 위해 첨부 텍스트를 인덱싱하면 된다. 
5단계. 법제처의 행정규칙 목록지원/검색 체계는 행정규칙, 보건복지부 조건으로 조회할 수 있으므로, 수집 후 제목 + 발령번호 기준으로 매칭해 두면 고시 연혁, 정식 전문 링크, 구조화된 식별자를 붙이기 한다. 
6단계. DB 스키마를 처음부터 나눠 설계하기 한다. 최소한 notice_list, notice_detail, attachment, law_mapping 네 테이블로 나누어야 한다. 
예를 들면:
notice_list: title, notice_no, doc_type, revision_type, posted_date, preview, list_url, detail_url
notice_detail: full_text, effective_date, department, raw_html
attachment: file_name, file_url, file_type, local_path, extracted_text
law_mapping: law_name, law_no, api_url, matched_score
이렇게 나누면 나중에 첨부 재파싱이나 법제처 재매칭을 다시 돌리기 쉽다. 이 단계는 공식 소스의 구조가 목록/상세/별첨으로 나뉘어 있기 때문에 특히 중요하다. 
7단계. 검색 인덱스를 따로 만들기로 하자. 사용자는 보통 고시번호, 제목 키워드, 본문 문장, 별표 용어로 찾기 때문에, 검색 인덱스는 최소 title + notice_no + preview + body_text + attachment_text를 묶어야 한다. 
8단계. 운영 단계에서는 RSS로 증분 갱신하기로 대응한다. 초기 1회는 게시판 전수 수집, 그다음부터는 RSS를 이용해 신규 글만 감시하는 방식이 가장 효율적이다. 복지부는 훈령/예규/고시/지침 RSS를 공식 제공하므로, 매일 또는 하루 2~4회 정도 폴링해서 새 글이 보이면 상세/첨부 수집만 추가하면 된다. 이렇게 하면 전체 재수집 없이 운영이 효율적이게 될 것이다. 

실제로는 이 순서로 진행하시면 됩니다.
1. 목록 전수 수집기 작성
2. 상세 페이지 수집기 작성
3. 첨부파일 URL 저장기 작성
4. PDF 텍스트 추출기 작성
5. 고시번호/제목 기반 검색 API 작성
6. 법제처 매핑 추가
7. RSS 증분 업데이트 붙이기

“모조리 수집”의 기준은 목록이 아니라 첨부까지 포함해야 완성도가 높아진다. 목록+상세만 모으면 “검색기”는 빨리 만들 수 있지만, 첨부까지 들어가야 “실무형 검색기”가 된다. 그래서 첨부까지 포함해야 한다. 

이제 첫 단계로 복지부 게시판 전수 수집용 파이썬 스크립트를 짜 주고, 다음 단계에 대한 가이드를 순차적으로 해줘야 한다.
